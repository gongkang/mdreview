# mdreview 轻量 Markdown 预览工具设计

日期：2026-05-22

## 背景

当前项目目录为空，尚无既有代码或架构约束。本设计从零定义一个轻量级 Markdown 预览工具，优先满足“命令行快速打开预览”的使用方式。

核心目标：

- 用户运行 `mdreview <file-or-directory>` 后快速进入预览。
- macOS 优先。
- 默认使用浏览器预览，后续可复用同一套 Web UI 做独立窗口模式。
- 首版坚持零配置，尽量依靠合理默认值和少量命令参数。
- 启动速度优先于功能堆叠。
- 本地服务默认只服务当前用户发起的预览会话，不暴露给局域网或其他来源。

## 产品范围

MVP 包含：

- 单文件和目录两种入口。
- 目录模式：左侧文件树、中间 Markdown 正文、右侧页面大纲。
- 单文件模式：隐藏左侧文件树，只保留正文和右侧大纲。
- GitHub Flavored Markdown：表格、任务列表、标题锚点等。
- 代码高亮。
- Mermaid 图表。
- 数学公式渲染。
- 单文件模式默认监听文件变化并自动刷新。
- 目录模式提供手动刷新，不做默认递归监听。

MVP 不包含：

- Markdown 编辑器能力。
- 项目级配置文件。
- 全文搜索。
- 导出 PDF/HTML。
- 主题系统。
- 独立桌面窗口。

这些功能可以作为后续阶段扩展，前提是不影响首版启动速度和核心预览体验。

## 推荐方案

采用 Node.js + TypeScript 实现 CLI、本地预览服务和浏览器端 UI。

CLI 启动本地 HTTP 服务并打开默认浏览器。服务端只负责安全读取本地文件、列出 Markdown 文件和推送单文件变化事件。Markdown 渲染主要在前端完成，使用成熟的 `unified` / `remark` / `rehype` 生态，并对 Mermaid 和数学公式做按需加载。服务端默认绑定 `127.0.0.1`，并用启动时生成的随机会话令牌保护本地文件 API。

该方案的主要理由：

- Markdown 插件生态成熟，首版实现成本低。
- 浏览器天然适合渲染文档、代码高亮、Mermaid 和公式。
- 本地服务启动轻，符合“启动快优先”。
- 后续若增加独立窗口模式，可复用同一套 Web UI。

## 命令行体验

核心命令：

```bash
mdreview <file-or-directory>
```

建议首版参数：

- `--port <number>`：指定端口。若未指定，自动选择空闲端口。
- `--no-open`：只启动服务并输出 URL，不自动打开浏览器。
- `--help`、`--version`：基础 CLI 信息。

不设计项目级配置文件。所有行为使用默认值和命令参数控制。

## 架构组件

### cli

负责解析命令参数、校验路径、判断启动模式、选择端口、启动服务，并打开浏览器。

路径判断规则：

- 文件路径：进入单文件模式。
- 目录路径：进入目录模式。
- 不存在、不可读、非文件也非目录：直接输出错误并退出。

### server

提供静态 Web UI 和本地文件 API。服务端需要保证所有文件访问都被限制在启动路径允许范围内，避免路径穿越。

建议 API：

- `GET /api/session`：返回当前会话模式、根路径显示名、默认文档。
- `GET /api/files`：目录模式下返回 Markdown 文件树。
- `GET /api/document?path=<relative-path>`：返回指定文档内容和元信息。
- `GET /api/events`：单文件模式下推送文件变化事件。

本地服务安全约束：

- 默认只绑定 `127.0.0.1`，不监听 `0.0.0.0`。
- 每次启动生成随机 session token，并把 token 放入打开的预览 URL fragment，例如 `/#token=<token>`。
- 前端从 URL fragment 读取 token，调用 `/api/*` 时通过请求头携带该 token；缺失或不匹配时返回 `401`。
- 不启用宽松 CORS；默认只接受同源请求。
- 文件访问必须基于启动根路径解析后的真实路径做范围校验，不能只依赖字符串前缀。

### watcher

只在单文件模式默认启用。监听目标文件变化，通过 Server-Sent Events 通知前端重新拉取内容。

目录模式首版不做默认递归监听，只提供手动刷新。这样可以避免大目录扫描和频繁文件事件影响启动速度。

### renderer-web

浏览器端应用，负责文档浏览器界面和 Markdown 渲染。

职责包括：

- 文件树渲染和文件选择。
- Markdown 正文渲染。
- 页面大纲生成和锚点跳转。
- 代码高亮。
- Mermaid 图表渲染。
- 数学公式渲染。
- 渲染错误的局部展示。
- 单文件自动刷新后的内容更新。

Mermaid 和数学公式相关依赖应按内容检测后动态加载，避免普通文档首屏加载过重。

Markdown HTML 安全策略：

- MVP 默认允许常见 Markdown 生成的 HTML，但必须经过 allowlist sanitization。
- 禁止 `script`、事件处理属性、`javascript:` URL、危险 iframe 和未知标签。
- Mermaid 使用严格安全模式；数学公式只使用渲染器生成的输出。两者都不能把未净化的用户输入直接写入 DOM。
- 如果某段原始 HTML 被过滤，正文继续渲染；必要时在开发日志中记录原因，界面不弹出打断式错误。

性能预算：

- 单文件模式目标是在典型 macOS 开发机上 800ms 内打印可访问 URL。
- 目录模式目标是 1200ms 内打开浏览器壳层，文件树可以在页面内异步加载。
- 首屏只加载基础 Markdown 渲染和代码高亮所需资源。
- Mermaid 和数学公式必须由内容检测触发动态加载，不进入普通文档的首屏关键路径。
- 目录扫描应默认跳过 `.git`、`node_modules`、`dist`、`build` 等常见重目录，保护启动和刷新体验。

### shared

保存 CLI、server、frontend 共用的 TypeScript 类型，例如：

- 会话模式。
- 文件节点。
- 文档内容响应。
- 结构化错误。
- 文件变化事件。

## UI 设计

目录模式采用三栏布局：

- 左侧：Markdown 文件树，仅展示 `.md` 和 `.markdown` 文件，并保留目录层级。扩展名匹配不区分大小写。
- 中间：正文预览区。
- 右侧：当前文档大纲，由 Markdown 标题生成。

单文件模式采用两栏布局：

- 中间：正文预览区。
- 右侧：当前文档大纲。

单文件模式不显示左侧文件树，因为没有可浏览的目录上下文。两种模式都应保持界面克制，避免变成完整编辑器。

## 数据流

启动流程：

1. 用户运行 `mdreview <path>`。
2. CLI 解析绝对路径，判断单文件或目录模式。
3. CLI 启动本地 server。
4. CLI 自动打开 `http://localhost:<port>`，或在 `--no-open` 下只输出 URL。
5. 前端请求 `/api/session` 初始化界面。
6. 目录模式请求 `/api/files`，单文件模式跳过文件树。
7. 目录模式默认打开根目录下的 README 文件，匹配顺序为 `README.md`、`README.markdown`，大小写不敏感；如果不存在，则打开按相对路径不区分大小写排序后的第一个 Markdown 文件。

打开文档流程：

1. 前端请求 `/api/document?path=<relative-path>`。
2. server 校验相对路径是否仍位于允许范围内。
3. server 读取文件，返回 `{ path, name, mtime, content }`。
4. 前端渲染 Markdown，并从标题节点生成右侧大纲。

刷新流程：

- 单文件模式：watcher 检测目标文件变化，推送 `document:changed`，前端重新请求当前文档并尽量保留滚动位置。
- 目录模式：用户点击刷新按钮后重新请求 `/api/files`。如果当前文档仍存在，则刷新内容；如果已删除，正文区显示文件不存在状态。

渲染流程：

1. 先运行基础 Markdown 管线，处理 GFM、表格、任务列表、代码块和标题锚点。
2. 对渲染结果执行 allowlist sanitization。
3. 检测 Mermaid 代码块后动态加载 Mermaid，并使用严格安全模式渲染图表。
4. 检测数学语法后动态加载公式渲染器，只插入渲染器生成的安全输出。
5. 单个图表或公式渲染失败时，只在对应块展示错误，不影响正文其他内容。

## 错误处理

CLI 错误应简短明确：

- 路径不存在。
- 路径不可读。
- 路径不是文件或目录。
- 用户显式指定的端口被占用。

端口默认自动选择空闲端口。只有用户显式传入 `--port` 且端口不可用时才报错。浏览器打开失败不终止服务，而是在终端输出 URL。

服务端 API 使用结构化错误：

```json
{
  "code": "FILE_NOT_FOUND",
  "message": "File not found"
}
```

前端错误分层：

- 会话初始化失败：显示全页错误。
- API token 缺失或不匹配：显示会话无效状态，并提示重新运行命令。
- 当前文档读取失败：正文区显示错误。
- Mermaid 或公式渲染失败：仅在对应块内显示错误，并保留原始内容。
- 单文件自动刷新遇到临时半截内容：保留上一次成功渲染结果，并提示等待下一次稳定更新。

## 测试策略

单元测试：

- CLI 参数解析。
- 单文件和目录模式判断。
- 路径访问范围校验。
- API token 校验。
- Markdown 文件扩展名识别。
- 文件树过滤。
- 默认 README 选择和排序规则。
- API 错误结构。
- Markdown 渲染管线的基础样例和 HTML sanitization。

集成测试：

- 启动真实 server，使用临时目录和测试 Markdown 文件。
- 覆盖 `/api/session`、`/api/files`、`/api/document`。
- 覆盖缺失或错误 token 时 API 返回 `401`。
- 覆盖路径穿越拦截。
- 覆盖文件删除后的响应。
- 覆盖显式端口占用时的错误行为。

浏览器测试：

- 目录模式能看到文件树、正文和大纲。
- 单文件模式隐藏左侧文件树。
- 单文件变化后页面刷新且不空白。
- Mermaid 和数学公式内容不会导致整页渲染失败。
- 带脚本或事件属性的 Markdown 不会执行脚本。
